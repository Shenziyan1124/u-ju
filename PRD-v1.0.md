## 一、需求背景及分析（为什么要做）

### 1.1 需求背景介绍

**需求来源**: 用户提出 / 商业用途

**背景描述**:
- **市场趋势**: 线下活动复苏，企业/机构对数字化活动管理工具需求增长
- **当前痛点**: 
  - 传统活动报名方式低效（微信群接龙、Excel统计）
  - 缺乏统一的票务管理和支付能力
  - 发布者和参与者之间缺乏高效连接渠道
- **为什么现在做**: 技术栈成熟（UniApp + Spring Boot），可快速构建MVP验证核心价值

**数据支撑**:
- 目标用户群: 企业HR、培训机构、会展公司、社团组织
- 预期覆盖: 初期本地测试，后续可扩展至区域级平台
- 对研发团队: 验证 AI-DLC Pipeline 全流程能力

### 1.2 产品目的（三角度分析）

#### 🎯 战略角度
- 构建活动管理领域的数字化基础设施
- 验证"三端统一"技术架构的可行性（小程序+H5+后台管理）
- 为后续扩展（评价系统、数据分析）奠定基础

#### 🔧 产品角度
- 提供端到端的活动管理解决方案（发布→报名→支付→核销）
- 差异化优势：原生质感UI设计、轻量级部署、快速上线
- 技术可行性：基于成熟开源框架，降低维护成本

#### 👤 用户角度
- **发布者**: 减少人工统计成本，提升报名转化率
- **参与者**: 一站式发现活动、便捷报名、电子票管理
- **管理员**: 可视化运营数据、高效审核流程

### 1.3 用户故事地图

| 用户角色 | 目标 | 行为步骤 | 痛点/期望 |
|:--------:|:-----|:---------|:----------|
| 参与者 | 发现并参加感兴趣的活动 | 浏览首页 → 搜索筛选 → 查看详情 → 填写报名 → 支付(如需) → 获取电子票 | 活动信息分散、报名流程繁琐、支付不便捷 |
| 发布者 | 高效发布和管理活动 | 创建活动 → 设置票型 → 审核报名 → 查看数据 | 统计工作量大、缺乏实时反馈、无法触达用户 |
| 管理员 | 平台运营和内容管控 | 审核认证 → 内容管理 → 数据监控 → 系统配置 | 信息分散、审核效率低、缺乏数据支撑 |

## 📌 假设与不确定性（Assumptions & Uncertainties）

### 我做出的假设

| # | 假设内容 | 置信度 | 来源/依据 | 需用户确认？ |
|---|---------|--------|-----------|-------------|
| A1 | 用户需要同时支持小程序和 H5 | 高 | 用户指定 UniApp | ❌ 已确认 |
| A2 | 后端使用 Java Spring Boot | 高 | 已实现的代码库 | ❌ 已确认 |
| A3 | 设计风格偏好"原生质感"而非"AI渐变风" | 高 | 用户明确要求 | ❌ 已确认 |
| A4 | MVP阶段不需要真实支付/短信服务 | 中 | 成本控制考虑 | ⚠️ 默认采用 |
| A5 | 单机部署即可满足初期需求 | 高 | MVP规模较小 | ⚠️ 默认采用 |

### 我识别出的歧义

| # | 歧义点 | 我的解释A | 我的解释B | 推荐 |
|---|-------|----------|----------|------|
| D1 | "后台管理"的权限范围 | 仅超级管理员 | 多角色（运营/客服/财务） | 先做A，B作为二期 |
| D2 | 活动审核是否必须 | 所有活动需审核 | 仅标记违规内容 | 先做A（安全优先） |

### 方案权衡（Tradeoffs）

| 方案 | 描述 | 优点 | 缺点 | 复杂度 | 🏆 推荐 |
|------|------|------|------|--------|:------:|
| A | 三端独立部署 | 解耦彻底 | 运维成本高 | 高 | ❌ |
| B | 前后端分离，后端统一 | 架构清晰，复用性好 | 需要API网关 | 中 | ✅ **已选** |
| C | Serverless | 免运维 | 调试困难，冷启动 | 高 | ❌ |

**选择理由**: 项目处于 MVP 阶段，方案B在复杂度和扩展性间取得平衡。

### 3.1 用户模块

#### 3.1.1 功能权限区分

| 用户角色 | 可访问功能 | 权限级别 | 识别方式 |
|:--------:|:----------:|:--------:|:--------:|
| 游客 | 浏览活动、查看详情 | 只读 | 未登录态 |
| 参与者 | 报名、收藏、消息、个人中心 | 读写 | JWT Token (PARTICIPANT) |
| 发布者 | 发布活动、管理报名、查看数据 | 读写 | JWT Token (ORGANIZER) |
| 管理员 | 全部功能 + 后台管理 | 管理 | JWT Token (ADMIN) |

#### 3.1.2 注册流程

```
[输入手机号] → [获取验证码] → [填写表单] → [提交注册]
                                            ↓
                                    ┌──────────────┐
                                    │ 校验通过?    │
                                    └──────┬───────┘
                                      是 ↓ 否
                              [创建用户] [错误提示]
                                   ↓
                              [自动登录]
                                   ↓
                              [跳转首页]
```

**验收标准 (DoD)**:
- [ ] 手机号格式校验（11位数字）
- [ ] 密码强度校验（≥8位，含大小写字母和数字）
- [ ] 短信验证码5分钟有效，60秒防重复发送
- [ ] 注册成功自动登录并跳转首页
- [ ] 错误场景有友好提示（手机号已注册、验证码错误等）

#### 3.1.3 登录方式

| 登录方式 | 流程 | 适用场景 |
|:--------:|:-----|:--------:|
| 密码登录 | 手机号 + 密码 → JWT Token | 常规登录 |
| 短信登录 | 手机号 + 验证码 → 自动注册/登录 | 忘记密码/新用户 |

#### 3.1.4 异常场景处理

| 异常场景 | 触发条件 | 用户提示 | 系统行为 |
|:--------:|:--------:|:--------:|:--------:|
| 手机号已注册 | 注册时检测 | "该手机号已被注册，请直接登录" | 跳转登录页 |
| 验证码错误/过期 | 登录/注册提交 | "验证码无效或已过期，请重新获取" | 刷新验证码 |
| 密码错误 | 登录提交 | "手机号或密码错误" | 不透露具体是哪个错 |
| 账户被禁用 | 登录成功后检测 | "账号已被禁用，请联系客服" | 清除Token |

### 3.2 活动模块

#### 3.2.1 活动发布（发布者）

**功能入口**: 个人中心 → "发布活动"

**界面字段**:

| 区域 | 展示字段 | 字段来源 | 数据转换规则 | 默认值 |
|:----:|:--------:|:--------:|:-------------|:------:|
| 基本信息 | 标题、封面图、分类、标签 | 用户输入 | 标题1-100字，标签≤5个 | 空 |
| 时间地点 | 开始/结束时间、截止时间、地点类型、地址 | 用户选择器 | 截止时间 < 开始时间 | 当前时间+1天 |
| 票务设置 | 是否免费、票型列表、人数限制 | 用户输入 | JSON序列化存储 | 免费，无限制 |
| 其他设置 | 审核模式、公开展示、联系方式 | 开关/文本 | Boolean/String | 自动通过、公开 |

**操作流程**:
```
[填写基本信息] → [设置时间和地点] → [配置票务] → [其他设置]
       ↓                  ↓                ↓              ↓
   [校验必填项]      [校验时间逻辑]   [校验价格]     [保存草稿/发布]
                                          ↓
                              ┌──────────────┐
                              │ 选择操作?    │
                              └──────┬───────┘
                            存草稿 ↓ 发布
                        [保存Draft]  [status=1, 审核队列]
```

**验收标准 (DoD)**:
- [ ] 所有必填项有明确的空值校验
- [ ] 时间逻辑正确（截止 < 开始 < 结束）
- [ ] 付费活动必须至少设置一个票型
- [ ] 草稿状态的活动不可被普通用户看到
- [ ] 发布后的活动进入待审核状态（如开启审核）

### 4.3 移动端专项规范（v1.1 新增）

| 场景 | 行为 | CSS 实现 |

| 内容不足一屏 | **不出现滚动条** | `page { height: 100%; overflow-y: auto; }` |
| 内容超过一屏 | 正常滚动 | 同上，框架自动处理 |
| 固定底部栏 | 始终可见 | `position: fixed; bottom: 0;` |

#### 设计风格约束（Anti-Patterns）

| ❌ 禁止 | ✅ 要求 | 原因 |
|:-------:|:-------:|:-----:|
| `linear-gradient` 按钮 | 纯色填充 `background: #1677ff` | 避免 AI 生成感 |
| `box-shadow` 卡片 | 细线边框 `border: 1rpx solid` | 原生质感 |
| 紫色系主色 | Ant Design 蓝 `#1677ff` | 符合国内用户习惯 |
| 圆角 > 20rpx | ≤20rpx | 移动端紧凑感 |
